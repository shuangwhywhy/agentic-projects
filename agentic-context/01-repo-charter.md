# Repo 1 Charter

## Repo 1 的定位

Repo 1 是整个大项目的 `Project Context Repo`。  
它是上下文与控制面，不是系统实现面。

这个仓库的核心价值不是“存文档”，而是把项目的认知秩序、工作入口、上层约束和 handoff 逻辑固定下来，使后续任何 Agent 都能快速知道：

- 项目到底要做什么
- 现在做到哪一层了
- 哪些东西已经确定
- 哪些东西还未确定
- 自己当前应该做什么
- 自己当前不应该做什么

## Repo 1 与 Repo 2 的明确分工

### Repo 1

负责：

- 项目抽象与工程哲学
- 产品定义与 PRD
- 跨层语义基准维护
- hierarchy 的固定
- 上下文资产沉淀
- Agent 入口、阅读顺序与工作纪律
- 对 Layer 3/4 的边界、输入条件与 handoff 约束

不负责：

- 最终系统架构设计
- 代码、配置、运行时、测试与部署工件
- 把架构或实现影子误包装成已经定稿的结果

### Repo 2

负责：

- 基于 Repo 1 稳定输入开展真实架构设计
- 基于架构设计开展实现、测试、脚本、配置、运行工件生产
- 在具体工作项内使用 execution stages 推进问题

不负责：

- 反向改写 Repo 1 的上位原则与产品定义
- 以局部实现便利为理由推翻 Layer 1/2

## 四层 hierarchy 在 Repo 1 中的职责

### Layer 1: Concept / Philosophy

Repo 1 必须真正把这一层做出来。  
它负责定义系统是什么、为什么这样设计、哪些边界不可协商、判断优先级如何排列。

### Layer 2: Product / PRD

Repo 1 必须真正把这一层做出来。  
它负责定义系统面向谁、解决什么问题、有哪些核心能力、有哪些产品边界、交付形式是什么。

### Layer 3: Architecture

Repo 1 不负责完成这一层的最终结果。  
Repo 1 在这里提供的是：

- 输入前提
- 约束条件
- 必须回应的问题
- 进入条件
- handoff 逻辑

### Layer 4: Implementation

Repo 1 不负责完成这一层的最终结果。  
Repo 1 在这里提供的是：

- 实现进入条件
- 追溯要求
- 验证要求
- 工作流边界
- 与 Repo 1 的回路关系

## 项目总秩序

本项目的总秩序是：

1. 先以 hierarchy 理解项目
2. 先把 Layer 1/2 稳定下来
3. 再为 Layer 3/4 提供明确入口
4. 再在 Repo 2 中对具体任务使用 execution stages

因此：

- hierarchy 先于 stage
- context/control 先于 realization
- 上层约束先于下层优化
- 验证性 progress 先于表面动作量

## Repo 1 需要长期守住的五个控制目标

1. 防止后续 Agent 接入后自由漂移
2. 防止后续工作越级推翻上层
3. 防止原始提示词中的混合内容继续混层
4. 防止项目重新退化成 execution-stage 混合仓库
5. 防止 Repo 2 在没有稳定输入时被迫盲目开工

## Repo 1 的稳定输出形式

Repo 1 应长期维护五类稳定资产：

1. 上位原则资产
2. 产品定义资产
3. 跨层语义资产
4. handoff 资产
5. 决策与开放项治理资产

其中前三类是本仓库的实质建设重点，后两类是本仓库对后续工作的控制接口。

## 变更规则

- 若要改动 Layer 1，必须说明为什么原有上位原则无法继续成立
- 若要改动 Layer 2，必须说明这会如何影响用户目标、能力边界与 Repo 2 的承接
- 若是 Layer 3/4 发现上层冲突，优先提交上推问题，而不是在下层私自重写上层
- 若只是具体任务的局部推进需求，应留在 Repo 2，通过 stage workflow 处理
