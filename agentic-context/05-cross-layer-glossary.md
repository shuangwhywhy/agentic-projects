# Cross-Layer Glossary

本文件是 Repo 1 的跨层公共语义基准。  
它不是某一层的附属说明，而是 Layer 1 / Layer 2 / Layer 3 / Layer 4 共同引用的 canonical 语义来源。

## 文档目的与权威性说明

本 glossary 的目标是：先钉语义，再做建模。

它负责在 Repo 1 中先固定：

- 核心术语的主名与英文别名
- 术语的语义类型
- 术语的一句话定义
- 术语的边界与相邻关系
- 术语的 canonical 引用规则
- 术语对 Layer 3 与 Layer 4 的约束
- 高风险混淆项的禁止解释方式

一旦本文件定义了某个术语，后续 Layer 1 / Layer 2 / Layer 3 / Layer 4 默认不得各自重定义。  
下层只能在不改变 semantic core 的前提下补充 operational context。

若本文件与其他文档的术语解释发生冲突，以本文件为准；若本文件本身不足以支撑后续工作，应先回到 Repo 1 补本 glossary，再继续下层工作。

## 覆盖范围与非目标

### 本轮覆盖范围

本轮只覆盖：

- 核心词
- 必要邻接词
- 足以支撑后续 layer 稳定展开的最小闭合集合

本轮核心词包括：

- 项目
- 当前有效目标
- 目标版本
- 任务单元
- progress
- 待验证进度
- 正式进度
- token 分类
- 监管事件
- 纠偏动作

本轮必要邻接词包括：

- 验证
- 验收
- 冻结
- 返工
- 归因
- 目标变更

### 非目标

本 glossary 明确不负责：

- 对象字段设计
- JSON schema 或其他 wire shape
- 状态机设计
- 存储与索引设计
- API 设计
- 代码结构与运行时结构
- 完整 ontology 铺陈

本 glossary 不替 Layer 3 做对象建模。  
它只规定：这些词在建模前是什么意思，什么不算它们。

本 glossary 不替 Layer 4 做实现对象化。  
它只规定：实现不得为了便利改写这些词的上位语义。

## 术语类型规则

每个术语在被讨论前，必须先明确语义类型。  
本 glossary 默认使用以下类型集合：

- `对象`
  - 指在项目中可被追踪、承接或引用的稳定语义单元
- `度量与状态口径`
  - 指用于计算、汇报、比较或区分推进状态的口径，而不是对象本身
- `分类体系`
  - 指对某类东西进行分类的规则体系，而不是一次具体归类结果
- `事件`
  - 指某个发生过、可被记录、可触发后续处理的语义发生点
- `动作`
  - 指为推进、验证、修复、归因或治理而执行的动作
- `其他必要类型`
  - 仅在前述类型不够时使用，且必须显式说明原因

当前 glossary 使用了一个必要扩展类型：

- `状态标签`
  - 说明：有些词本质上是在描述一个被赋予对象或流程的状态标签，既不是对象，也不是动作或事件；例如“冻结”

禁止在未声明类型的情况下直接解释术语。  
也禁止把术语类型在不同层之间悄悄切换。

## Canonical 命名与引用规则

- 每个术语只有一个 canonical main name
- 除显式例外外，默认以中文主名为准，英文别名用于与 Repo 2 的架构与代码命名衔接
- 别名只能作为兼容表达，不得与主名并列争夺定义权
- 首次在其他文档中出现 glossary 术语时，应优先使用 canonical main name
- 如需别名，只能作为括注或兼容说明出现
- 若后续文档使用非 canonical 说法，必须能明确映射回 glossary 主名
- 若映射不成立，应先补 glossary，而不是在下层文档里私自造义

### 当前已固定的 canonical 项

- `当前有效目标` 是 canonical main name；`当前目标` 只能作为受限别名
- `progress` 是当前 Repo 1 中保留的 canonical 例外主名；中文“进度”可作解释性对照，但不替代该主名
- `待验证进度` 与 `正式进度` 是同一 progress 体系下的不同口径，不是两个平行对象
- `token 分类` 只回答“按什么维度分类”，不回答“某次实际归因结果是什么”

## Glossary Entries

### 项目

- 主名：项目
- 英文别名：`Project`
- 术语类型：对象
- 一句话定义：项目是本系统中承接一个复杂任务推进周期的最高语义边界，用于统一目标、任务、组织、资源、监管与进度语境。
- 边界：
  - 包含：当前有效目标、目标版本链、任务单元体系、组织结构、token 相关记录、监管事件与纠偏动作的共同上下文
  - 不包含：单个任务单元、单次对话、单次运行进程、单个仓库路径
- 关系：
  - 项目承载当前有效目标
  - 项目包含任务单元体系与 progress 口径
  - 项目也是监管事件与纠偏动作的归属边界
- 别名与禁用混用项：
  - 可接受兼容说法：项目上下文
  - 禁止混用：仓库、工作区、会话、单次运行
- Canonical 引用规则：
  - 当表达最高语义边界时，必须使用“项目”
  - 不得用“repo”“workspace”或“session”替代项目语义
- 对 Layer 3 的约束：
  - Layer 3 必须把项目视为统一语义边界，不能把它压扁成一个任务对象
- 对 Layer 4 的约束：
  - Layer 4 可以把项目映射到实现中的工作实体，但不能把文件夹、进程或数据库行直接等同于项目语义本身

### 当前有效目标

- 主名：当前有效目标
- 英文别名：`Current Effective Goal`
- 术语类型：对象
- 一句话定义：当前有效目标是在当前时刻实际支配项目推进、任务判断和治理优先级的目标对象。
- 边界：
  - 包含：当前被采纳、当前用于指导推进的目标内容与约束
  - 不包含：全部历史目标版本、尚未采纳的新提议、单次用户临时措辞变化
- 关系：
  - 当前有效目标由某个目标版本承载并处于当前生效位置
  - 当前有效目标约束任务单元、progress 与监管判断
  - 目标变更事件可能导致当前有效目标切换
- 别名与禁用混用项：
  - 受限别名：当前目标
  - 禁止混用：目标版本、目标变更、用户最新一句话
- Canonical 引用规则：
  - 在需要表达“当前正在治理项目的目标”时，必须使用“当前有效目标”
  - “当前目标”只能在不引起歧义时作为括注或兼容说法
- 对 Layer 3 的约束：
  - Layer 3 不能把当前有效目标与目标版本合并成同一个语义对象
- 对 Layer 4 的约束：
  - Layer 4 不得把最新输入、最新草稿或 UI 当前选中项自动当成当前有效目标

### 目标版本

- 主名：目标版本
- 英文别名：`Goal Version`
- 术语类型：对象
- 一句话定义：目标版本是对目标在某一时点的版本化表达，用于记录目标内容、变化来源和与其他版本的关系。
- 边界：
  - 包含：某一版目标内容及其版本身份
  - 不包含：当前是否生效的治理判断、目标变更动作本身
- 关系：
  - 当前有效目标由某个目标版本承载
  - 目标变更事件连接目标版本之间的演化关系
- 别名与禁用混用项：
  - 可接受兼容说法：goal version
  - 禁止混用：当前有效目标、目标变更事件
- Canonical 引用规则：
  - 只要强调“版本化表达”，就使用“目标版本”
  - 不得把“当前有效目标”写成“当前目标版本”来偷换语义
- 对 Layer 3 的约束：
  - Layer 3 必须保留版本与生效判断的语义分离，不能仅用一个“最新目标字段”替代
- 对 Layer 4 的约束：
  - Layer 4 可以有版本数据结构，但不得用“最后一次编辑”直接代替目标版本语义

### 任务单元

- 主名：任务单元
- 英文别名：`Task Unit`
- 术语类型：对象
- 一句话定义：任务单元是可比较、可验证、可纳入统一 progress 口径的最小工作语义单元。
- 边界：
  - 包含：清晰的完成条件、验证要求与归属关系
  - 不包含：粗粒度任务树节点、消息回合、提交次数、临时想法
- 关系：
  - 任务单元受当前有效目标约束
  - 任务单元是 progress 统计与验证动作的基本承载对象
  - 返工、冻结与纠偏动作都可能作用于任务单元
- 别名与禁用混用项：
  - 可接受兼容说法：最小任务单元
  - 禁止混用：任务树、子任务、消息轮次、实现步骤
- Canonical 引用规则：
  - 凡涉及 progress 统计、验证口径、实现追溯，优先使用“任务单元”
- 对 Layer 3 的约束：
  - Layer 3 必须把任务单元做成可比较、可验证的语义单位，不能退化成“任何干过的事都算”
- 对 Layer 4 的约束：
  - Layer 4 的实现项若不能映射到任务单元，就不能直接计入正式 progress

### progress

- 主名：progress
- 中文对照：进度
- 英文别名：`Progress`
- 术语类型：度量与状态口径
- 一句话定义：progress 是建立在任务单元与明确计入口径上的统一推进度量，不是活动量本身。
- 边界：
  - 包含：按定义口径累计或说明的推进结果
  - 不包含：消息数、思考轮次、提交次数、表面忙碌程度
- 关系：
  - progress 以任务单元为统计基础
  - 待验证进度与正式进度是 progress 体系下的不同口径
- 别名与禁用混用项：
  - 解释性对照：进度
  - 禁止混用：活动量、提交量、轮次数、消息量
- Canonical 引用规则：
  - Repo 1 中涉及统一进度口径时，使用 `progress`
  - 中文“进度”只能作为说明性对照，不能替代 canonical 主名
- 对 Layer 3 的约束：
  - Layer 3 必须先定义 progress 口径，再设计统计方法
- 对 Layer 4 的约束：
  - Layer 4 不得把任何实现活动默认折算为 progress

### 待验证进度

- 主名：待验证进度
- 英文别名：`Pending-Verification Progress`
- 术语类型：度量与状态口径
- 一句话定义：待验证进度是已发生推进但尚未完成验证、因此尚不能正式计入口径的 progress 子口径。
- 边界：
  - 包含：已形成候选推进，但还未通过验证的部分
  - 不包含：纯草稿、无任务归属活动、已经正式计入的进度
- 关系：
  - 待验证进度属于 progress 体系
  - 通过验证后可转入正式进度；未通过则可能回到返工或冻结
- 别名与禁用混用项：
  - 可接受兼容说法：pending progress
  - 禁止混用：正式进度、活动量
- Canonical 引用规则：
  - 用于表达“已有推进候选，但尚未被正式确认”的口径时，必须使用“待验证进度”
- 对 Layer 3 的约束：
  - Layer 3 不能把待验证进度当成独立对象或最终累计结果
- 对 Layer 4 的约束：
  - Layer 4 不得因为产物已生成就把它直接视为正式进度

### 正式进度

- 主名：正式进度
- 英文别名：`Formal Progress`
- 术语类型：度量与状态口径
- 一句话定义：正式进度是在既定 progress 体系下，已经满足正式计入条件的推进口径。
- 边界：
  - 包含：已达到既定计入口径、可被正式汇报的推进结果
  - 不包含：待验证进度、纯草稿、未经确认的活动量
- 关系：
  - 正式进度属于 progress 体系
  - 它通常建立在任务单元的已验证推进之上
- 别名与禁用混用项：
  - 可接受兼容说法：formal progress
  - 禁止混用：待验证进度、活动量
- Canonical 引用规则：
  - 在需要明确“正式可计入”的口径时，使用“正式进度”
- 对 Layer 3 的约束：
  - Layer 3 必须定义正式进度与待验证进度的口径区别，而不是把它们做成两个平行对象
- 对 Layer 4 的约束：
  - Layer 4 只能在满足既定条件后计入正式进度，不能以“已运行”替代“已正式计入”

### token 分类

- 主名：token 分类
- 英文别名：`Token Taxonomy`
- 术语类型：分类体系
- 一句话定义：token 分类是对 token 消耗或使用进行分类的规则体系，用于后续统计、归因与优化。
- 边界：
  - 包含：分类维度与分类规则
  - 不包含：某次具体归因结果、某张统计报表、某个告警结论
- 关系：
  - token 分类为归因与统计提供前置分类基础
  - 分类体系本身不等于归因动作或归因结论
- 别名与禁用混用项：
  - 可接受兼容说法：token taxonomy
  - 禁止混用：token 归因结果、token 仪表盘、单次成本结论
- Canonical 引用规则：
  - 只要表达“分类规则体系”，就使用“token 分类”
  - 若表达的是“某次归因结果”，必须另用“归因”等术语
- 对 Layer 3 的约束：
  - Layer 3 必须先区分分类体系与归因结果，再做审计模型
- 对 Layer 4 的约束：
  - Layer 4 不得把前端展示分组或数据库枚举值自动等同为 token 分类全语义

### 监管事件

- 主名：监管事件
- 英文别名：`Supervision Event`
- 术语类型：事件
- 一句话定义：监管事件是监管与合规模块识别、记录或触发的可追溯治理事件，用于标记偏差、风险、异常或需要处理的监管节点。
- 边界：
  - 包含：已发生并被记录的监管触发点
  - 不包含：纠偏动作本身、长期状态标签、单条普通日志
- 关系：
  - 监管事件可能触发纠偏动作、冻结、返工或升级
  - 监管事件与证据、责任判断和影响分析相关
- 别名与禁用混用项：
  - 可接受兼容说法：compliance / supervision event
  - 禁止混用：纠偏动作、证据、告警级别、长期状态
- Canonical 引用规则：
  - 需要表达“监管侧已发生且可追溯的事件”时，必须使用“监管事件”
- 对 Layer 3 的约束：
  - Layer 3 必须把监管事件与纠偏动作、状态标签分开
- 对 Layer 4 的约束：
  - Layer 4 不得把所有日志、提示或弹窗都自动算作监管事件

### 纠偏动作

- 主名：纠偏动作
- 英文别名：`Corrective Action`
- 术语类型：动作
- 一句话定义：纠偏动作是为修正偏差、控制风险或恢复推进路径而采取的明确治理动作。
- 边界：
  - 包含：被要求、被规划或已执行的纠偏步骤
  - 不包含：监管事件本身、最终状态、纯提醒文本
- 关系：
  - 纠偏动作通常由监管事件触发
  - 返工可以是纠偏动作的一种具体形式
- 别名与禁用混用项：
  - 可接受兼容说法：corrective action
  - 禁止混用：监管事件、冻结状态、返工结果
- Canonical 引用规则：
  - 需要表达“为了纠正而采取的动作”时，必须使用“纠偏动作”
- 对 Layer 3 的约束：
  - Layer 3 不能把纠偏动作建模成事件或永久状态
- 对 Layer 4 的约束：
  - Layer 4 可以实现纠偏流程，但不得用单个布尔标记替代纠偏动作全语义

### 验证

- 主名：验证
- 英文别名：`Verification`
- 术语类型：动作
- 一句话定义：验证是检查某项任务单元或候选推进是否满足既定完成条件与证据要求的动作。
- 边界：
  - 包含：对完成条件、证据、结果一致性的检查
  - 不包含：最终业务采纳决定、长期状态变化
- 关系：
  - 验证作用于任务单元与待验证进度
  - 验证结果影响是否可进入正式进度
- 别名与禁用混用项：
  - 可接受兼容说法：verification
  - 禁止混用：验收、正式进度
- Canonical 引用规则：
  - 涉及“确认是否满足完成条件”的动作时，使用“验证”
- 对 Layer 3 的约束：
  - Layer 3 需要区分验证动作与 progress 口径本身
- 对 Layer 4 的约束：
  - Layer 4 不得把“代码能运行”直接等同为已完成验证

### 验收

- 主名：验收
- 英文别名：`Acceptance`
- 术语类型：动作
- 一句话定义：验收是对已完成或已验证结果进行正式接纳、确认或关闭处理的动作。
- 边界：
  - 包含：基于角色责任的正式接纳或关闭判断
  - 不包含：纯技术检查、纯自动化断言本身
- 关系：
  - 验收可以建立在验证之后，但两者不等同
  - 验收结果可能影响正式进度、阶段关闭或交付确认
- 别名与禁用混用项：
  - 可接受兼容说法：acceptance
  - 禁止混用：验证、监管事件
- Canonical 引用规则：
  - 当需要表达“正式接纳或关闭”的动作时，使用“验收”
- 对 Layer 3 的约束：
  - Layer 3 不得把验证与验收折叠成单一概念，除非明确说明只是局部流程简化
- 对 Layer 4 的约束：
  - Layer 4 不得把测试通过自动视作验收完成

### 冻结

- 主名：冻结
- 英文别名：`Freeze`
- 术语类型：状态标签
- 一句话定义：冻结是加在目标、任务单元或推进路径上的状态标签，表示其在当前阶段不得继续自动推进，需等待外部判断或后续处理。
- 边界：
  - 包含：被标记为暂停推进、暂停计入或暂停自动流转的状态
  - 不包含：监管事件本身、纠偏动作本身、永久废弃
- 关系：
  - 冻结可能由监管事件触发，也可能作为纠偏动作之后的状态结果
  - 冻结常与返工、升级或重新判断联动
- 别名与禁用混用项：
  - 可接受兼容说法：freeze
  - 禁止混用：监管事件、纠偏动作、删除
- Canonical 引用规则：
  - 当表达“被暂停继续自动推进的状态标签”时，使用“冻结”
- 对 Layer 3 的约束：
  - Layer 3 需要把冻结与事件、动作分开，不能混成一个节点
- 对 Layer 4 的约束：
  - Layer 4 可以实现冻结标记，但不得把“未处理”或“失败”自动等同于冻结

### 返工

- 主名：返工
- 英文别名：`Rework`
- 术语类型：动作
- 一句话定义：返工是对未满足要求或已被判定需要修正的任务单元重新开展工作的动作。
- 边界：
  - 包含：回到任务单元或相关产物上做修复与重做
  - 不包含：发现问题的事件、返工后的最终状态
- 关系：
  - 返工可以是纠偏动作的一种具体形式
  - 返工常与验证、冻结或正式进度回退相关
- 别名与禁用混用项：
  - 可接受兼容说法：rework
  - 禁止混用：纠偏动作总称、失败状态
- Canonical 引用规则：
  - 当表达“重新做某项工作”时，使用“返工”
- 对 Layer 3 的约束：
  - Layer 3 不得把返工直接建模成状态标签
- 对 Layer 4 的约束：
  - Layer 4 不得把任何代码修改都默认视为返工，只有在任务语义上回到修正路径时才算

### 归因

- 主名：归因
- 英文别名：`Attribution`
- 术语类型：动作
- 一句话定义：归因是将某项消耗、结果、风险或偏差关联到其来源范围、责任来源或影响路径的分析动作。
- 边界：
  - 包含：解释“来自哪里、落在哪个范围、由谁承担”
  - 不包含：预先定义的分类体系、单纯展示报表
- 关系：
  - 归因可使用 token 分类，但不等于 token 分类
  - 归因与监管事件、纠偏动作、资源消耗分析都有关系
- 别名与禁用混用项：
  - 可接受兼容说法：attribution
  - 禁止混用：token 分类、统计报表、责任结论全文
- Canonical 引用规则：
  - 需要表达“把某结果或消耗解释到来源”的动作时，使用“归因”
- 对 Layer 3 的约束：
  - Layer 3 必须把归因动作与分类体系分开
- 对 Layer 4 的约束：
  - Layer 4 不得把数据库字段填充或图表分组直接当成完整归因

### 目标变更

- 主名：目标变更
- 英文别名：`Goal Change`
- 术语类型：事件
- 一句话定义：目标变更是导致目标内容、目标优先级或当前有效目标发生调整的事件。
- 边界：
  - 包含：已发生的目标调整事实
  - 不包含：目标版本对象本身、未来候选提案、当前有效目标对象本身
- 关系：
  - 目标变更可能生成新目标版本，并可能改变当前有效目标
  - 目标变更会影响任务单元、progress 与监管判断
- 别名与禁用混用项：
  - 可接受兼容说法：goal change
  - 禁止混用：目标版本、当前有效目标
- Canonical 引用规则：
  - 当表达“目标发生了变化”这一语义发生点时，使用“目标变更”
- 对 Layer 3 的约束：
  - Layer 3 不能把目标变更事件与目标版本对象合并处理
- 对 Layer 4 的约束：
  - Layer 4 不得把任意文本编辑自动视作目标变更，除非它进入了项目治理意义上的变化链条

## 高风险混淆项清单

以下混淆在后续文档中被明确禁止：

- 把 `项目` 当成仓库、工作区、会话或单次运行
- 把 `当前有效目标` 当成 `目标版本`
- 把 `目标版本` 当成 `目标变更`
- 把 `任务单元` 当成任务树节点、消息轮次或实现步骤
- 把 `progress` 当成活动量、轮次数、消息数或提交数
- 把 `待验证进度` 与 `正式进度` 当成两个独立对象
- 把 `token 分类` 当成归因结果、仪表盘视图或单次成本分析
- 把 `监管事件` 当成 `纠偏动作`
- 把 `纠偏动作` 当成长期状态
- 把 `冻结` 当成事件或动作
- 把 `返工` 当成失败状态
- 把 `验证` 当成 `验收`
- 把 `验收` 当成单纯测试通过

## 跨层约束与变更规则

- Layer 1 可以继续补强原则，但不得绕开 glossary 术语重写 semantic core
- Layer 2 不得一边写 PRD 一边私自改 glossary 词义；若产品表达需要新术语，应先补 glossary
- Layer 3 不得一边建模一边私自改义；建模只能承接 glossary，不能反向给 glossary 下定义
- Layer 4 不得以实现便利替换 glossary 的高层定义；实现若需要更细命名，只能是下位命名，不得改 semantic core

若要修改 glossary 中已有术语，必须同时说明：

- 为什么现有定义不足
- 会影响哪些上层或下层文档
- 哪些 canonical 引用需要同步修改
- 是否会影响 Layer 3 / Layer 4 的既有承接方式

## 使用与验收提醒

本 glossary 完成后的最低验收标准是：

- 核心词与必要邻接词都有唯一 canonical entry
- 每个 entry 都有类型、边界、关系、禁用混用项和跨层约束
- README / AGENTS / layer 文档都把本 glossary 作为引用源
- Repo 1 中不再存在同名词的隐含竞争定义
- 本 glossary 没有越位进入 schema、状态机、存储或 API 设计
