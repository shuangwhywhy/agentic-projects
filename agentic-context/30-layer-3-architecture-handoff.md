# Layer 3: Architecture Handoff

本文件不是最终架构设计。  
它定义的是：当工作进入架构层时，Repo 1 应向 Repo 2 提供什么输入、约束与进入规则。

本文件中的核心术语以 [05-cross-layer-glossary.md](./05-cross-layer-glossary.md) 为准。  
Layer 3 只能在 glossary 已固定的语义上做对象建模，不能反向替 glossary 下定义。

所有未决治理项、临时假设和正式决定的当前状态，以 [06-open-questions-register.md](./06-open-questions-register.md) 为准。  
本文件可以引用这些治理项，但不能新建平行未决列表。

Repo 1 -> Repo 2 的实际 handoff 控制接口以 [07-repo1-to-repo2-handoff-packet-template.md](./07-repo1-to-repo2-handoff-packet-template.md) 为准。  
本文件负责定义架构承接边界，`07` 负责把稳定输入打包给 Repo 2。

## Layer 3 在 Repo 1 中的职责

Repo 1 对 Layer 3 的职责不是“替你设计架构”，而是“限制架构不走偏”。

因此本层只做四件事：

1. 指明架构工作的前置输入
2. 指明架构必须回应的问题
3. 指明架构不可越过的边界
4. 指明何时要把问题上推回 Layer 1/2

## 架构开始前必须承认的固定输入

来自 Layer 1：

- 系统面向复杂任务推进
- 动态目标是一等公民
- progress 必须基于最小任务单元与验证结果
- token 优化不得压倒真实性与合规
- 四大核心模块不可缺失
- 自组织必须可解释、可审计、可监管、可回退

来自 Layer 2：

- 产品是 Native 的 CLI + GUI 并存系统
- 用户默认通过少数 endpoints 与系统交互
- 默认用户表面只承认 `项目对接端点` 与 `监管升级端点`
- `项目对接端点` 负责汇报、澄清、确认、交付与日常同步
- `监管升级端点` 负责风险同步、偏差举证、合规提醒、升级与异常通知
- owner / manager / liability 型职能必须存在
- owner / manager / liability 型治理角色不等于默认用户端点
- 产品必须对项目、当前有效目标、任务、progress、token、组织、监管形成可理解表面

这些输入不是建议，而是架构前提。

## 架构层必须回答的问题

### 1. 真相模型

- 项目、当前有效目标、目标版本、任务单元、任务序列、progress、组织、token、监管事件之间的核心对象关系是什么

### 2. 状态与流转

- 目标变化后，任务、组织、资源、监管如何联动
- 任务单元如何流转，以及在验证前后、返工中或冻结时如何变化

### 3. 模块边界

- 四大核心模块如何分工
- 哪些能力是独立模块，哪些能力是跨模块协作

### 4. 组织治理

- 如何在责任结构上承接 `项目对接端点` 与 `监管升级端点`
- 如何让 owner / manager / liability 等非端点治理角色与默认双端点共存
- 如何保证监管升级通道不被项目对接表面吞掉
- 组织如何在不同复杂度和风险等级下伸缩

### 5. 审计与监管闭环

- token 如何采集、归因、聚合、告警
- 偏差如何被识别、举证、升级、纠偏、关闭

### 6. 交互一致性

- CLI 和 GUI 如何共享同一上下文真相与同一审计真相
- 用户直连特定 agent 时，如何不破坏统一治理结构

## 架构层禁止跳过的设计对象

- 动态目标版本化
- 最小任务单元标准
- progress 计算口径
- token 分类与归因模型
- 用户对接 endpoints
- owner / manager / liability 职能落点
- 监管证据与纠偏机制
- 组织扩张、收缩、回退与升级路径

## 架构层禁止行为

- 禁止把 execution stages 当成总架构结构
- 禁止把“agent 之间互相聊天”当成组织治理
- 禁止省略 token 审计与监管模块
- 禁止把 progress 简化成消息数、回合数、提交数
- 禁止用静态流程覆盖动态目标问题
- 禁止为了实现便利删掉用户对接 endpoints 或 owner 职能
- 禁止把 `监管升级端点` 吞并成 `项目对接端点` 的附属能力

## 何时必须上推

出现以下情况时，架构工作必须上推，而不是私自决定：

- 发现 Layer 1 原则之间有真实冲突
- 发现产品边界不足以支持关键架构判断
- 发现 glossary 现有术语不足以承接建模问题
- 发现新的开放项尚未进入 register
- 发现某项能力会改变用户与系统的责任关系
- 发现为了实现某种结构，必须削弱“真实 progress / token 审计 / 监管纠偏”中的核心要求

## 架构层在 Repo 2 中应产出的内容类型

Repo 2 的 Layer 3 至少应形成：

- 核心对象模型
- 模块边界说明
- 关键状态机与状态流
- 组织职责矩阵
- token 审计模型
- 监管与合规流程
- CLI / GUI 与系统真相的映射策略
- 开放项如何进入实施决策

这些是 Repo 2 的工作，不应在 Repo 1 被伪装成已完成结果。

## 进入 Layer 3 的判定条件

只有当以下条件成立，才建议进入 Repo 2 做架构：

- Layer 1 已足够稳定
- Layer 2 已足够说明产品目标与边界
- 关键开放项已经被标注，不再混在原则层里
- 当前任务明确属于“系统结构设计”，而不是“继续定义项目是什么”
