# Original Prompt Intake

本文件的目的，不是复述原始提示词，而是说明它应如何被吸纳、归位、分离与重组，成为 Repo 1 的有效输入。

原始来源：`/Users/yizhouqiang/Documents/ai/prompts/agents/多智能体系统设计.md`

## 对原始提示词的总体判断

原始提示词是高价值原始输入，不能忽略，也不能原样继承为项目结构。

它同时包含了四种不同性质的内容：

1. `concept/philosophy`
2. `product/PRD`
3. `architecture shadow`
4. `implementation shadow`

它的问题不在于质量低，而在于这些内容被写在了同一套推进提示中，彼此没有被层级化安置。

因此 Repo 1 的工作不是推翻它，而是把它重新放回正确的位置。

## 吸纳原则

### 1. 吸纳

保留原始提示词中的稳定、高优先级、可跨实现阶段继承的核心要求。

### 2. 归位

把每条要求判断为更适合进入 Layer 1、Layer 2，还是只作为 Layer 3/4 的输入前提。

### 3. 分离

把 execution-stage 思维和 project hierarchy 分开。

### 4. 重组

围绕 Repo 1 的真实职责，重新组织成“上位资产 + handoff 接口”，而不是继续沿用混合提示结构。

## 需要被强吸纳的核心要求

以下要求应视为原始提示词中最稳定、最应继承的部分：

- 系统面向复杂任务，不是静态问答
- 目标是动态的，任务、组织、资源与监管都要随之联动
- progress 必须建立在最小任务单元与验证结果之上
- token 成本必须被记录、归因、分析与优化
- 系统必须同时具备四大核心模块：
  - 目标与任务管理
  - 组织与 agent 管理
  - 资源与 token 审计管理
  - 监管与合规模块
- 系统必须有面向用户的对接 endpoints
- 系统必须存在 owner / manager / liability 型职能
- 监管必须覆盖过程与结果，且要能监管 owner
- 用户没有逐条列出的问题，只要对完整设计关键，就应被主动识别与补全
- 低 token 成本必须服从真实有效推进与持续合规

这些内容已经被转译进 Layer 1 和 Layer 2，不再依赖原始提示词的单次阅读。

## 原始提示词内容的大致归位

### 更适合进入 Layer 1 的内容

- “真实完成复杂任务”优先于多智能体概念展示
- 动态目标是一等公民
- progress 只能建立在验证过的最小任务单元上
- 低成本不能压倒真实性、验证、审计、监管
- 自组织必须可解释、可审计、可监管、可回退
- 用户是最终责任人与最终决策者
- 不得编造能力、状态、证据、验证结果
- 必须主动补全完成系统所必需的 open questions

### 更适合进入 Layer 2 的内容

- Native 工具定位
- CLI + GUI 并存
- 面向复杂任务推进的产品形态
- 默认少数用户对接 endpoints，必要时允许直连指定 agent
- 四大系统级能力必须在产品层被显式看见
- 用户应能看见项目、目标、任务、进度、token、组织、监管等视图

### 更适合进入 Layer 3 的内容

- 目标对象模型、目标版本化
- 任务单元模型、依赖关系、状态流转
- progress 口径与验证闭环
- token 采集点、分类、统计维度、归因逻辑
- 组织拓扑、角色矩阵、升级路径
- 监管事件、偏差证据、纠偏流程、告警与归档

这些不应在 Repo 1 中被伪装成最终架构，而应被转成架构层必须回应的问题与约束。

### 更适合进入 Layer 4 的内容

- 代码结构
- 配置
- 测试
- runtime artifacts
- scripts
- 埋点实现
- dashboard 实现
- 交互端点的具体协议和界面细节

这些只应作为 Repo 2 的实现议题，不应在 Repo 1 中越位展开。

## 需要被重述而不是照搬的部分

原始提示词中有一些内容更像“对单次模型运行的控制指令”，而不是项目本体事实，例如：

- “你是首席架构师”的单轮角色设定
- `/version` 运行规则
- 对当前执行提示词的保密要求

这些内容有价值，但它们属于 prompt-runtime control，而不是 Repo 1 的项目结构本身。

在 Repo 1 中，它们最多只能被转译为：

- Agent 入口纪律
- 上下文使用规则
- 交付一致性要求

不能直接占据项目层级结构。

## Repo 1 对原始提示词的重组结果

Repo 1 现在对原始提示词做了五件事：

1. 把项目愿景和上位原则沉淀为 Layer 1
2. 把产品目标和能力边界沉淀为 Layer 2
3. 把核心术语收敛为跨层 glossary，作为公共语义基准
4. 把架构影子改写为 Layer 3 的 handoff 输入
5. 把实现影子改写为 Layer 4 的 handoff 输入

这意味着后续 Agent 不再需要依赖“读完整份原始提示词后自行揣测项目结构”，而是可以直接从 Repo 1 获得分层后的稳定入口。

## 这份 intake 文件的作用

它负责持续提醒后续工作：

- 原始提示词是要求来源，不是最终目录
- 高价值内容必须继承，但要按层归位
- 混层内容必须拆开，不得继续混用
- 核心术语必须先被 glossary 钉住，再进入下层建模与实现
- Repo 1 的使命是控制上下文，不是复制提示词
