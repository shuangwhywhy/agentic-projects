# Layer 4: Implementation Handoff

本文件不是实现方案。  
它定义的是：实现层什么时候可以开始、必须继承什么、必须验证什么、不能擅自改写什么。

本文件中的核心术语以 [05-cross-layer-glossary.md](./05-cross-layer-glossary.md) 为准。  
Layer 4 只能在 glossary 已固定的语义上做实现对象化，不能以实现便利改写 semantic core。

所有未决治理项、临时假设和正式决定的当前状态，以 [06-open-questions-register.md](./06-open-questions-register.md) 为准。  
本文件只能引用这些治理项，不能自行维护平行未决列表。

## Layer 4 在 Repo 1 中的职责

Repo 1 对 Layer 4 的职责不是“提前写代码”，而是保证实现工作不会脱离项目的上位秩序。

## 实现开始前的必备输入

Layer 4 不应直接从原始提示词起步。  
它必须继承：

- 跨层 glossary 的核心语义
- Layer 1 的上位原则
- Layer 2 的产品定义
- Layer 3 的正式架构输出

若缺少其中任一项，实现都容易退化成局部最优或假性推进。

## 实现层必须保留的追溯链

任何实现工作都应能回答：

- 它在实现哪一项产品能力
- 它受哪一条上位原则约束
- 它依赖哪一个架构对象或流程
- 它将如何被验证
- 它将如何影响 progress、token 审计或监管证据

如果回答不了，说明实现与项目控制面已经脱钩。

## 实现层必须兑现的最小结果类型

这些结果应在 Repo 2 中出现，而不是在 Repo 1 中提前伪造：

- 代码
- 配置
- 测试
- scripts
- telemetry / audit instrumentation
- runtime artifacts
- 验证报告

## 实现层工作规则

### 1. execution stages 可以使用，但只在局部任务内使用

在 Repo 2 中，具体任务当然可以按 `research / design / implementation / review` 推进。  
但这些 stages 只是实现工作流，不得反向替代项目 hierarchy。

### 2. 每个实现项都必须能映射到任务单元

实现项不能只是“写了一些东西”，而必须能映射到：

- 对应任务单元
- 对应完成标准
- 对应验证动作
- 是否计入正式 progress

### 3. token 与监管相关能力不能后补成装饰

token 审计和监管合规不是“系统做完后再加的监控层”，而是实现时就必须考虑的系统能力。

### 4. 用户对接 endpoints 不能被实现简化掉

若实现层为了省事，把用户端点、责任归属、升级路径等产品结构删掉，则属于越界。

## 实现层必须通过的验证视角

至少应验证：

- 动态目标变化后，系统是否仍能更新任务、组织与监管判断
- progress 是否只在验证后被正式累计
- token 是否能区分用户交互、内部通信、试错与有效推进
- 用户是否能通过 endpoints 理解当前项目状态
- 监管是否能给出偏差证据、影响范围和纠偏建议
- owner / manager / liability 职能是否真实存在，而不是名义标签

## 实现层的禁止行为

- 禁止跳过 Layer 3，直接从产品想象写代码
- 禁止把埋点、审计、验证、监管全部留到后补阶段
- 禁止把“已运行”误当成“已验证”
- 禁止把局部自动化成功误当成系统真实 progress
- 禁止为了节省 token 或开发时间，删除关键审计和纠偏能力

## 实现层与 Repo 1 的回路

实现过程中若出现以下情况，应回到 Repo 1：

- 发现产品定义不够支撑实现判断
- 发现 glossary 术语不足以支撑实现承接
- 发现新的开放项尚未进入 register
- 发现架构承接不了 Layer 1/2 的关键约束
- 发现原有开放项已经影响实现边界
- 发现某项实现会改变用户责任、监管逻辑或 progress 口径

回到 Repo 1 的目的不是暂停，而是让上位结构先被修正，再继续实现。

## 进入 Layer 4 的判定条件

只有当以下条件成立时，才应进入 Repo 2 的实现工作：

- Layer 3 已形成正式架构输入
- 当前任务已经被拆成可验证的实现项
- progress 口径已知
- token 审计与监管要求已被纳入实现范围
- 实现工作不再承担“定义项目是什么”的责任
