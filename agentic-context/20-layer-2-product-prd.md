# Layer 2: Product / PRD

本层负责把 Layer 1 的上位哲学转译成产品定义与 PRD 原型。

本文件中的核心术语以 [05-cross-layer-glossary.md](./05-cross-layer-glossary.md) 为准。  
本文件可以补充产品语境，但不改写 glossary 中已经固定的 semantic core。

本文件中的开放问题只保留摘要。  
其当前状态、owner、next action 与来源 / 依据 统一以 [06-open-questions-register.md](./06-open-questions-register.md) 为准。

## 产品定义

这是一个 `Native` 的复杂任务多智能体系统产品。  
默认形态为 `CLI + GUI 并存`：

- CLI 用于高频控制、透明调度、脚本化推进、调试与高级操作
- GUI 用于项目视图、任务树、进度、token 仪表盘、组织拓扑、监管告警、证据与审批

产品目标不是回答问题，而是帮助用户把复杂任务真实推进到可交付结果。

## 核心用户与关系模型

### 用户

- 是最终责任人
- 是最终决策者
- 是最终验收者
- 默认通过少数 endpoints 与系统交互

### 系统

- 扮演乙方团队
- 负责拆解、组织、推进、汇报、解释、纠偏、复盘
- 默认不把所有内部 agent 暴露给用户

### 特殊模式

- 当用户明确要求时，可以直连特定 agent
- 但直连不改变总责任秩序，owner / manager / liability 职能仍然必须存在

## 产品要解决的问题

用户不是缺一个“会聊天的 agent”，而是缺一个能在复杂、动态、多约束条件下，持续推进项目的系统。

具体来说，产品要解决：

- 目标会变，但系统不能失去推进方向
- 任务很复杂，必须持续拆解、重排、验证
- 组织不是固定的，必须按复杂度和风险动态调整
- token 会被大量消耗，必须可归因、可分析、可优化
- 偏差和失真会在过程里出现，必须尽早发现并纠正
- 用户不可能事先列完所有关键问题，系统必须主动补全关键缺口

## 产品边界

### In Scope

- 复杂任务项目管理与推进
- 动态目标版本管理
- 任务单元化与 progress 管理
- 多 agent 组织治理
- token 与资源审计
- 过程期监管与合规纠偏
- 用户对接 endpoints
- 验证、复核、升级与交付机制

### Out of Scope

- 单纯的聊天机器人体验优化
- 仅展示多 agent 角色的演示系统
- 脱离任务推进的通用社交型 agent
- 在 Repo 1 中直接落定架构或实现细节

## 产品核心能力

### 1. 目标与任务管理

产品必须能让用户和系统共同维护：

- 当前有效目标
- 目标版本与变更原因
- 目标变化影响到的任务范围
- 任务拆解、任务单元、依赖关系与优先级
- progress 以及其验证依据

### 2. 组织与 agent 管理

产品必须能表达：

- 默认用户对接 endpoints
- owner / manager / liability 职能
- 执行型与复核型能力
- 组织扩张、收缩、重组与升级路径

### 3. 资源与 token 审计管理

产品必须能让用户看见并追溯：

- 用户输入 token
- AI 输出 token
- 内部 agent 通信 token
- 试错 token
- 有效推进 token
- 成本趋势、浪费来源、效率变化

### 4. 监管与合规模块

产品必须能持续检查：

- 当前推进是否偏离用户意图
- 是否违反系统高优先级规则
- 是否出现虚假、越权、职责缺位、误解、资源浪费
- 偏差证据是什么
- 应如何纠偏，是否需要升级给用户

## 用户应看到的产品表面

无论 CLI 还是 GUI，用户都应能看懂系统在做什么，而不是只看到内部 agent 的闲聊。

至少应有以下产品表面对象：

- 项目
- 当前有效目标与目标版本
- 任务树与最小任务单元
- progress 及其验证依据
- 当前组织结构与责任归属
- token 成本与效率
- 监管告警与证据
- 决策待办与待确认项

## 默认交互模型

### 默认用户对接角色集的正式决定

本段是 Layer 2 的 canonical 产品决定表达，用于正式关闭 `OQR-001`。  
它关闭的不是命名偏好，而是默认最小用户对接角色集、合并与分离边界，以及非端点治理角色与默认用户端点的关系边界。

- 默认用户表面只承认两个 endpoints：
  - `项目对接端点`
  - `监管升级端点`
- `项目对接端点` 的职责范围固定为：
  - 汇报
  - 澄清
  - 确认
  - 交付
  - 日常同步
- `监管升级端点` 的职责范围固定为：
  - 风险同步
  - 偏差举证
  - 合规提醒
  - 升级
  - 异常、冻结与返工通知
- `直连指定 agent` 不是默认 endpoints 集的一部分，只属于扩展模式
- `owner / manager / liability` 型治理角色继续存在，但不得被写成“都等于一个用户端点”
- `项目对接端点` 可以承接 owner / manager 的默认对外协同表面，但不等于全部治理角色
- `监管升级端点` 是独立默认通道，不能被“统一项目负责人”之类表述吞并

### 默认模式

- 用户默认与 `项目对接端点` 和 `监管升级端点` 交互
- 前者承担项目层对接，后者承担监管升级通道

### 扩展模式

- 用户可按需连接指定 agent
- 但系统仍需保留统一的任务、责任、审计与监管闭环

## 产品级成功标准

本产品至少应做到：

- 复杂任务能被持续拆解并推进
- 目标变化不会导致系统失焦
- progress 不再建立在粗粒度活动量上
- token 成本可以被解释、比较、优化
- 用户能看见系统为什么这样推进
- 偏差能在过程期被发现并纠正
- 用户不需要逐条枚举所有关键问题，系统仍能继续成形

## 产品级非目标

- 不追求“每个内部 agent 都有华丽人格”
- 不追求以最低 token 为唯一优化目标
- 不追求在 Repo 1 中直接给出架构图或代码框架
- 不追求用单个全能 agent 替代组织治理

## 当前 PRD 原型给 Layer 3 的输入

Layer 3 在 Repo 2 中至少必须承接这些产品问题：

- 系统如何表达当前有效目标、目标版本与目标变更
- 系统如何定义最小任务单元与 progress 口径
- 系统如何在默认双端点固定输入下组织 owner、非端点治理角色、执行能力与复核能力
- 系统如何采集和解释 token 成本
- 系统如何做过程期监管、证据归因与纠偏
- CLI 和 GUI 如何共享同一项目真相，而不是各自漂移
