# Layer 1: Concept / Philosophy

本层定义项目的上位认知，不讨论实现技巧，也不提前替架构或代码做决定。

## 系统是什么

这是一个面向复杂任务推进的多智能体系统项目。  
它不是静态问答系统，不是单轮工作流，不是几个 agent 的角色表演。

系统的根本目标是：

在动态目标、动态任务、动态组织、动态资源约束和动态监管条件下，持续推进真实、可验证、可追溯、持续合规的任务完成。

## 上位哲学

### 1. 任务真实性优先

系统的第一目标不是“显得很聪明”，而是“真的推进任务”。

### 2. 动态目标优先

目标变化不是异常，而是系统必须显式承接和治理的事实。

### 3. Progress 必须可验证

只有经过验证、可回溯、有完成依据的任务推进，才能算正式 progress。

### 4. 粒度一致性先于统计

在统计进度之前，必须先建立最小任务单元与验收口径。

### 5. 低 token 是从属目标

token 优化必须服务于真实有效推进，不能以牺牲验证、审计、监管和纠偏为代价。

### 6. 自组织必须受治理

组织可以动态生成与调整，但必须可解释、可审计、可监管、可回退。

### 7. 监管是过程能力

监管与合规不是交付末端的“验收动作”，而是全过程持续工作的纠偏机制。

### 8. 不完整输入不等于不重要

只要用户已经给出了稳定的大框架、核心目标与价值排序，系统就必须主动识别关键 open questions，并将其纳入设计。

### 9. 上层先于下层

Concept/Philosophy 决定 Product/PRD 的边界，Product/PRD 决定 Architecture 的问题空间，Architecture 决定 Implementation 的承接方式。

### 10. Context 本身是生产资产

项目上下文不是附属说明，而是影响后续判断质量、方向一致性和组织纪律的核心资产。

## 不可协商边界

以下边界不可被下层以便利性为由绕开：

- 系统必须面向复杂任务推进，而非静态答复
- 目标、任务、组织、资源、监管都必须承认动态性
- 四大核心模块不可缺失
- 用户对接 endpoints 不可缺失
- owner / manager / liability 型统筹职能不可缺失
- token 审计不可缺失
- 监管与合规不可缺失，且必须能监管 owner
- progress 不能由表面活动量代替
- 不得编造能力、状态、证据、结果
- 必须主动补全关键开放问题

## 判断优先级

当项目出现判断冲突时，按以下顺序处理：

1. 用户明确目标、边界与约束
2. Layer 1 的上位原则
3. Layer 2 的产品定义
4. Layer 3 的架构设计
5. Layer 4 的实现策略
6. 局部 execution stages 中的临时方案

## 对“有效推进”的定义

有效推进至少同时满足：

- 与当前有效目标一致
- 有明确任务单元归属
- 有完成标准
- 有验证结果
- 可解释为何计入 progress
- 未违反高优先级约束

否则只能算活动量、尝试量、返工量或待验证产物。

## 对“低成本”的定义

低成本不是“少做事”，而是：

- 减少无效试错
- 减少偏航
- 减少不可追溯沟通
- 减少无监管的返工
- 提高每单位 token 对真实 progress 的贡献

## 对“多智能体”的定义

多智能体不是多个名义角色同时存在，而是：

- 存在职责分化
- 存在组织治理
- 存在责任归属
- 存在协作与复核关系
- 存在资源与监管机制
- 存在面向任务推进的闭环

如果缺少这些，只是“多会话”或“多角色包装”，不是本项目要的系统。

## Layer 1 对下层的硬约束

Layer 2 必须把这些原则转译成清晰产品定义。  
Layer 3 必须解释如何承接这些原则，而不是削弱它们。  
Layer 4 必须通过实现把这些原则变成可观测、可验证、可审计的行为。

若下层无法承接，应上推问题，而不是绕开原则。
